Foundational Technical Review (FTR)이 현재 일시 중단되었습니다.

Foundational Technical Review (FTR)이 현재 일시 중단되었습니다.

FTR 신청을 최대한 빨리 재개할 수 있도록 적극적으로 작업 중이며, 신청 제출이 가능해지면 알려드리겠습니다. 불편을 끼쳐드려 죄송합니다. FTR은 AWS Marketplace에서 제품이나 솔루션을 등록하거나 판매하는 데 필수 요구사항이 아닙니다. 자세한 내용은 다음을 참조하세요 AWS Marketplace 판매자 설명서.

소개

기초 기술 검토(FTR)는 AWS 파트너의 솔루션을 보안, 성능, 운영 프로세스 등 아마존 웹 서비스(AWS)의 최선의 사례들에 대해 평가합니다. 이것들은 고객의 성공에 가장 중요합니다. FTR을 통과하는 것은 AWS Competency와 AWS Service Ready 등의 AWS 파트너 네트워크(APN) 프로그램에 AWS 소프트웨어 파트너 자격을 부여하는 데 필요하지만, 기술 솔루션을 제공하는 모든 AWS 파트너는 AWS 파트너 센트럴을 통해 FTR 검토를 요청할 수 있습니다.

이 체크리스트는 파트너가 AWS에서 호스팅하는 솔루션(예: SaaS 솔루션)에 적용됩니다. 모든 중요한 애플리케이션 구성 요소는 AWS에서 호스팅되어야 합니다. 만약 당신의 솔루션이 이 요구사항을 충족하지 못한다면, Foundational Technical Review Checklist Index을 참조하십시오. 콘텐츠 전송 네트워크(CDNs)나 도메인 이름 시스템(DNSs), 또는 기업 신원 제공자와 같은 엣지 서비스에 대해 외부 제공자를 사용할 수 있습니다.

이 체크리스트는 또한 일본어, 포르투갈어, 스페인어로도 제공되고 있습니다.

기대하는 사항들

AWS 파트너는 FTR 요청을 제출하기 전에 이 문서와 FTR 가이드를 상세히 검토해야 합니다. 이 문서에 대해 궁금한 점이 있다면, 파트너 개발 대표(PDR) 또는 파트너 개발 관리자(PDM)에게 연락하십시오. AWS는 언제든지 이 문서에 변경을 가할 권리를 보유하고 있습니다.

FTR 요청은 AWS 파트너 센트럴을 사용하여 제출해야 합니다. 요청을 제출하는 방법에 대한 자세한 정보는 AWS 기초 기술 검토를 선택하고 FTR 요청을 클릭하십시오.

또한 대체 수동 프로세스를 사용하여 FTR을 요청할 수도 있습니다. 자세한 정보는 AWS 파트너 센트럴의 FTR 가이드를 참조하십시오.

AWS 파트너 센트럴을 통해 FTR 검토를 위해 제공을 제출한 후, AWS 파트너 솔루션 아키텍트(PSA)는 제출한 문서를 검토하고 이메일로 연락을 드릴 것입니다. FTR 자가 평가가 제출되고 모든 요구 사항이 충족되면, FTR이 승인됩니다. 만약 모든 요구사항이 충족되지 않는다면, AWS PSA는 수정할 사항 목록과 FTR을 완성하는 방법에 대한 안내를 제공할 것입니다. PSA에게 모든 수정 사항을 실행하고 확인을 제공한 후에 FTR이 승인됩니다. 만약 6개월 이내에 FTR 결과를 수정하고 FTR 승인을 받지 못하면, 새로운 FTR 검토를 위해 제공을 제출해야 하며, 이는 시간이 지남에 따라 검증 체크리스트의 변경으로 인해 추가적인 제어가 필요할 수 있습니다. 갱신의 경우, 현재 FTR 만료일 전에 FTR을 완료하지 못하면, 검토 중인 제공이 파트너 솔루션 파인더(PSF)에서 목록에서 제거됩니다.

하나 이상의 제공에 대해 승인된 FTR이 없다면, 계정은 AWS 소프트웨어 경로에서 Validated+ 단계에서 제거됩니다. 소프트웨어 경로에서 레벨이 내려가면, APN 고객 참여(ACE) 및 AWS ISV 가속 프로그램에 대한 접근을 포함한 Validated 단계의 혜택을 잃게 됩니다.

다음 항목들은 AWS에 의해 기술 검증이 실행되기 전에 검증될 것입니다.

  1. 1.0기초 기술 검토 요구 사항

    1. 1.1소프트웨어 경로 멤버십

      AWS 파트너는 Software Path에 등록되어 있어야 합니다. AWS 시스템 통합(SI) 파트너는 PDR/PDM과 대화하여 Software Path에 가입하는 방법을 알아볼 수 있습니다.

Partner-hosted FTR requirements

다음 요구 사항은 AWS 파트너가 자신의 AWS 계정에서 호스팅하고 운영하는 모든 컴포넌트에 적용됩니다. CIS 벤치마크 1.4 ID를 포함하는 컨트롤은 자동화된 AWS 계정 스캔에 의해 처리됩니다. 그것들에 대해 자가 평가에서 응답할 필요는 없습니다.

사전 요구 사항

  • CIS - Center for Internet Security (CIS) 벤치마크 보고서

    인터넷 보안 센터 (CIS) 벤치마크 보고서는 자동화된 CIS 스캔을 통해 워크로드를 호스팅하는 AWS 계정의 보안 제어를 검증합니다.이 보고서는 업계 표준 보안 벤치마크를 준수함을 보여줍니다.선택 사항이지만 FTR 검토 프로세스를 신속하게 진행하려면 완성된 CIS 벤치마크 보고서를 FTR 요청 양식과 함께 Partner Central에 제출하는 것이 좋습니다.가능한 경우 보고서를 제출물에 첨부해 주십시오.

  • GA - 일반 구매 가능 여부

    FTR에 제출된 모든 솔루션은 다음 중 하나여야 합니다.

    • 일반 구매 가능 (GA) 또는
    • 고객이 구매할 수 있습니다.

파트너 호스팅

  • HOST-001 - 호스팅 모델을 확인합니다.

    FTR의 자격을 얻으려면 제품이 AWS에서 완전히 실행되어야 합니다. 여기에는 애플리케이션 및 제어 평면이 포함됩니다. 애플리케이션 평면은 판매자의 AWS 계정, 구매자의 AWS 계정 또는 둘 다에서 실행할 수 있습니다. 자세한 내용은

    Control plane vs. application plane 백서를 참조하십시오. 애플리케이션 데이터를 전송, 저장 또는 처리하는 데 제품이 사용하는 타사 서비스도 콘텐츠 전송 네트워크(CDN), 도메인 이름 시스템(DNS) 및 기업 ID 공급자(IdP)를 제외하고 AWS에서 완전히 실행되어야 합니다.

    제품이 보안, 모니터링, 데이터 복제 또는 마이그레이션에 사용하는 에이전트 또는 게이트웨이는 AWS 외부의 구매자 소유 환경에서 실행할 수 있으며, 온프레미스를 포함하지만 저장 및 분석을 위해 데이터를 AWS로만 전송해야 합니다.

    검토를 위해 아키텍처 다이어그램을 포함해야 합니다. 자세한 내용은 Architecture diagram을 참조하십시오.

    Marketplace에 나열하는 것은 FTR을 완료하는 데 필요하지 않습니다.

지원 레벨

  • SUP-001 - 모든 프로덕션 AWS 계정에 대해 AWS 비즈니스 지원 계층(또는 그 이상)을 구독하거나, AWS 지원이 필요한 문제를 처리하는 행동 계획을 가지고 있어야 합니다.

    모든 AWS 프로덕션 계정에 대해 AWS 비즈니스 지원 계층 또는 그 이상을 구독하는 것이 권장됩니다. 자세한 정보는 AWS 지원 계획 비교를 참조하십시오. 프리미엄 지원이 없는 경우, AWS 지원이 필요한 문제를 처리하는 행동 계획이 있어야 합니다. AWS 지원은 성능 최적화, 비용 절감, 빠른 혁신을 능동적으로 돕기 위해 도구와 기술, 인력, 프로그램의 혼합을 제공합니다. AWS Business Support는 AWS Trusted Advisor와 AWS Personal Health Dashboard에 대한 접근 및 더 빠른 응답 시간을 포함한 추가 혜택을 제공합니다.

아키텍처 리뷰

  • WAFR-001 - 매년 최소 한 번씩 아키텍처 검토를 수행합니다.

    AWS 특정 모범 사례가 포함된 문서화된 아키텍처 표준을 사용하여 생산 워크로드의 주기적인 아키텍처 검토를 수행합니다 (최소한 매년 한 번). AWS 워크로드에 대한 내부 표준이 있으면 이러한 검토에 사용하는 것이 좋습니다. 내부 표준이 없는 경우, AWS Well-Architected Framework를 사용하는 것이 좋습니다.

  • WAFR-002 - AWS Shared Responsibility Models for Security and Resiliency를 검토합니다.

    AWS Shared Responsibility Model for Security와 AWS Shared Responsibility Model for Resiliency를 검토합니다. 제품의 아키텍처와 운영 프로세스가 이 모델에서 정의된 고객의 책임을 다루는지 확인합니다.

아키텍처 및 운영 제어

AWS Well-Architected Review (WAFR) 또는 SOC2 유형 II 감사를 완료한 경우 아래 규제 항목 대부분을 우회할 수 있습니다.아키텍처 검토 또는 감사는 AWS 직원, 웰아키텍티드 파트너 프로그램 (WAPP) 파트너 또는 자격을 갖춘 SOC2 감사자가 완료해야 합니다.이 프로세스를 이미 완료한 경우 FTR 대체 검증 체크리스트 에서 추가 정보를 참조하십시오.

다음 요구 사항은 AWS 파트너가 자체 AWS 계정에서 호스팅 및 운영하는 모든 구성 요소에 적용됩니다.인터넷 보안 센터 (CIS) 벤치마크 5.0 ID를 포함하는 제어 기능은 자동 AWS 계정 스캔에 포함됩니다.CIS 벤치마크 v3.0을 제출하는 경우 자체 평가 시트에 해당 규제 항목이 언급되어 있습니다. 자세한 내용은 각 버전의 CIS 요구 사항에 대한 컨트롤 매핑 을 참조하십시오.

AWS 루트 계정

다음 요구 사항은 루트 사용자에 대한 비밀번호를 설정한 경우에만 적용됩니다. AWS Organizations를 사용하여 AWS 계정을 생성하고 루트 사용자 비밀번호를 요청하지 않는 경우, 이 요구 사항들이 적용되지 않습니다.

  • ARC-001 - 예외적인 경우에만 루트 사용자를 사용합니다.

    루트 사용자는 계정과 그 리소스에 무제한 접근 권한이 있으므로 예외적인 경우에만 사용하면 AWS 리소스를 보호하는 데 도움이 됩니다. AWS 루트 사용자는 일상적인 작업, 심지어 관리 작업에도 사용되어서는 안됩니다. 대신, best practice of using the root user only to create your first AWS Identity and Access Management (IAM) user을 준수하십시오. 그런 다음 루트 사용자 자격 증명을 안전하게 보관하고 몇 가지 계정 및 서비스 관리 작업만 수행하는 데 사용하십시오. 루트 사용자로 로그인해야 하는 작업을 보려면 AWS Tasks That Require Root User를 참조하십시오. FTR은 루트 사용에 대한 지속적인 모니터링을 필요로 하지 않습니다.

    AWS Organizations를 사용하여 AWS 루트 계정 액세스를 중앙 집중화하는 것이 좋습니다. AWS Organizations를 사용하여 새 멤버 계정을 생성하고 루트 액세스를 중앙 집중화하는 경우, 조직의 모든 멤버 계정에 대한 루트 사용자 자격 증명을 삭제하도록 선택할 수 있습니다. 루트 액세스를 중앙에서 관리하는 방법에 대한 자세한 내용은여기에서 확인할 수 있습니다.

  • ARC-004 - 루트 사용자에 대한 액세스 키를 제거합니다. (CIS_5.0 1.3/Security Control IAM.4 또는 CIS_3.0 1.4/Security Control IAM.4)

    AWS API에 대한 프로그래밍 방식의 접근은 절대 루트 사용자를 사용해서는 안됩니다. 루트 사용자에 대한 정적 액세스 키를 생성하는 것은 좋지 않습니다. 이미 존재하는 경우, 해당 키를 사용하는 모든 프로세스를 AWS Identity and Access Management (IAM) 역할의 임시 액세스 키, 또는 필요한 경우 IAM 사용자의 정적 액세스 키를 사용하도록 전환해야 합니다.

    멤버 계정으로 AWS Organizations를 사용하고 루트 액세스를 중앙 집중화할 계획이 없는 경우, 루트 사용자는 AWS 계정의 특정 작업에만 사용해야 하므로 루트 사용자의 액세스 키를 제거하는 것이 중요합니다.

  • ARC-005 - 루트 계정 자격 증명 무단 사용에 대한 사고 대응(IR) 런북을 작성합니다.

    루트 계정 자격 증명 무단 사용에 대한 적절한 대응을 상세하게 기술한 runbook은 루트 사용자가 침해당한 경우 즉시 행동할 수 있도록 합니다. 루트 계정 자격 증명이 접근 불가능한 경우, Unable to Sign in & Submit Billing or Account Request,하거나 로그인 불가능 및 청구 또는 계정 요청 제출 페이지를 통해 계정 및 청구 지원에 연락해야 합니다. 런북의 예는 here에서 찾을 수 있습니다.

AWS로부터의 커뮤니케이션

  • ACOM-001 (CIS_5.0 또는 CIS_3.0 1.2/Security Control Account.1) - AWS 계정 연락처를 구성하십시오.

    계정이 AWS Organizations에서 관리되지 않는 경우, 대체 계정 연락처는 AWS가 필요한 경우 적절한 직원과 연락하는 데 도움을 줍니다. 계정의 alternate contacts를 개인이 아닌 그룹에 지정하십시오.

  • ACOM-002 - 회사가 소유한 이메일 주소와 전화 번호를 포함하여 계정 연락처 정보를 설정하십시오.

    연락처 정보에 회사가 소유한 이메일 주소와 전화 번호를 사용하면 해당 개인이 귀사와 더 이상 연관되지 않더라도 이를 접근할 수 있습니다.

Identity and Access Management

이 섹션의 요구 사항은 솔루션에 대한 신원 및 액세스를 관리하는 방법에 관한 것입니다. 이에는 AWS Identity and Access Management(IAM) 리소스뿐만 아니라 기타 신원 공급자와 액세스 메커니즘(예: 제품의 최종 사용자를 위한 액세스 관리, 데이터베이스 자격 증명 처리, 서비스 간 액세스 제어)도 포함됩니다.

  • IAM-001 - AWS 액세스가 있는 모든 인간 신원에 대해 다중 요소 인증(MFA) 활성화. (CIS_5.0 1.9/Security Control IAM.5 또는 CIS_3.0 1.10/Security Control IAM.5)

    AWS 계정에 접근하기 전에 모든 인간 신원이 MFA를 사용하여 인증해야 합니다. 일반적으로 이는 기업 신원 공급자 내에서 MFA를 활성화하는 것을 의미합니다. 기존의 레거시 IAM 사용자가 있으면 그 원칙에 대해 콘솔 액세스에 대해 MFA를 활성화해야 합니다. IAM 사용자에 대해 MFA를 활성화하면 추가 보안 계층이 제공됩니다. MFA를 사용하면 사용자는 고유한 인증 코드(일회용 비밀번호 또는 OTP)를 생성하는 장치를 가지게 됩니다. 사용자는 일반 자격 증명(사용자 이름과 비밀번호)과 OTP를 모두 제공해야 합니다. MFA 장치는 특별한 하드웨어일 수도 있고 가상 장치일 수도 있습니다(예: 스마트폰에서 앱으로 실행될 수 있음). 기계 신원에는 MFA가 필요하지 않다는 점을 주의하십시오.

  • IAM-002 - 정기적으로 자격 증명 회전. (CIS_5.0 1.13/Security Control IAM.3 또는 CIS_3.0 1.14/Security Control IAM.3)

    가능하면 role을 가정하여 얻은 임시 IAM 자격 증명을 사용하십시오.(CIS 1.14/Security Control IAM.3) IAM role의 사용이 불가능한 경우, 이런 자격 증명이 오용되는 위험을 줄이기 위해 다음과 같은 통제 조치를 구현하십시오.

    • IAM 액세스 키를 정기적으로 교체하십시오 (최소 90일마다 권장).
    • 모든 정적 키 인벤토리의 사용 위치를 관리하고 사용하지 않는 액세스 키는 제거하십시오.
    • AWS CloudTrail 로그를 모니터링하여 비정상적인 활동이나 기타 잠재적인 오용을 탐지하십시오.(예 : AWS GuardDuty 사용).
    • 오용을 탐지된 경우 자격 증명을 취소하는 실행 절차 또는 표준 운영 절차(SOP)를 정의하십시오.

    자세한 내용은 IAM 사용자의 액세스 키 관리 (https://docs.aws.amazon.com/IAM/latest/UserGuide/id_credentials_access-keys.html) 를 참조하십시오.

  • IAM-003 - 강력한 비밀번호 정책 사용. ([CIS_5.0 1.7/Security Control IAM.15 과 CIS_5.0 1.8/Security Control IAM.16] 또는 [CIS_3.0 1.8/Security Control IAM.15 과 CIS_3.0 1.9/Security Control IAM.16])

    강력한 비밀번호 정책을 강제하고, 사용자들에게 흔한 비밀번호나 재사용된 비밀번호를 피하도록 교육해야 합니다. IAM 사용자를 위해서는, IAM 콘솔의 계정 설정 페이지에서 계정의 비밀번호 정책을 만들 수 있습니다. 비밀번호 정책을 사용해 비밀번호 요구사항을 정의할 수 있습니다, 예를 들면 최소 길이와 비알파벳 문자가 필요한지 등등. 자세한 정보는 Setting an Account Password Policy for IAM users을 참조하십시오.

  • IAM-004 - AWS 액세스가 필요한 모든 사람에 대해 개별 신원 생성 (공유 자격 증명 없음).

    개별 엔티티를 생성하고 각 사용자에게 고유한 보안 자격 증명과 권한을 부여하세요. 개별 엔티티와 공유된 자격증명이 없으면 각 사용자의 활동을 감사할 수 있습니다.

  • IAM-005 - 제 3자에게 접근 권한을 제공하기 위해 IAM 역할 및 그 임시 보안 자격증명을 사용하세요.

    IAM 사용자를 생성하고 그 자격 증명을 귀사 외부의 사람들과 공유하지 마십시오. 귀하의 계정에 대한 AWS API 호출이 필요한 외부 서비스 (예를 들어, 귀하의 계정의 AWS CloudWatch 메트릭에 액세스하는 모니터링 솔루션)는 교차 계정 역할을 사용해야 합니다. 자세한 정보는 Providing access to AWS accounts owned by third parties을 참조하십시오.

  • IAM-006 - 최소 권한 원칙 적용.

    granting least privilege라는 표준 보안 조언을 따라야 합니다. 신원이 필요로 하는 접근만 허용하여 특정 조건 하에 특정 AWS 리소스에 대한 특정 행동에 대한 접근을 허용합니다. 개별 사용자에 대해 권한을 정의하는 대신 그룹과 신원 속성에 의존하여 권한을 대규모로 동적으로 설정하세요. 예를 들어, 그들의 프로젝트에 대한 리소스만 관리할 수 있는 접근을 개발자 그룹에 허용할 수 있습니다. 이렇게 하면, 개발자가 그룹에서 제거되면 그룹이 접근 제어를 위해 사용된 모든 곳에서 개발자의 접근이 취소되며, 접근 정책에 변경을 가하지 않아도 됩니다.

  • IAM-007 - 생명 주기에 기반한 접근을 관리하십시오.

    운영자와 애플리케이션 생명주기와 당신의 중앙 페더레이션 공급자와 IAM과 접근 제어를 통합하십시오. 예를 들어, 조직을 떠나거나 역할을 바꿀 때 사용자의 접근 권한을 제거하십시오.

  • IAM-008 - 신원을 분기별로 감사하십시오.

    당신의 신원 공급자와 IAM에 설정된 신원을 감사함으로써 오직 인가된 신원만이 당신의 작업에 접근할 수 있도록 도와줍니다. 예를 들어, 조직을 떠난 사람을 제거하고, 더 이상 필요하지 않은 cross-account 역할을 제거하십시오. IAM 엔티티에 의해 접근되는 서비스에 대한 권한을 주기적으로 감사하는 과정을 마련하십시오. 이것은 사용하지 않는 권한을 제거하기 위해 수정해야 할 정책을 확인하는 데 도움이 됩니다. 자세한 정보는, Refining permissions in AWS using last accessed information을 참조하십시오.

  • IAM-009 - 애플리케이션 코드에 자격 증명을 내장하지 마십시오.

    당신의 애플리케이션에서 사용하는 모든 자격 증명 (예를 들어, IAM 액세스 키와 데이터베이스 비밀번호)은 절대 애플리케이션의 소스 코드에 포함되거나 어떤 방식으로든 소스 제어에 커밋되어서는 안 됩니다.

  • IAM-010 - 비밀 정보를 안전하게 저장하십시오.

    비밀 정보를 전송 중이거나 저장 중일 때 모두 암호화하고, 특정 신원에게만 접근을 허용하는 세분화된 접근 제어를 정의하고, 비밀 정보에 대한 접근을 감사 로그에 기록하십시오. 우리는 AWS Secrets Manager, AWS Systems Manager Parameter Store, 또는 AWS 파트너 솔루션과 같은 목적에 맞는 비밀 관리 서비스를 사용하는 것을 권장하지만, 이러한 요구사항을 충족하는 내부적으로 개발된 솔루션도 허용됩니다.

  • IAM-011 - 모든 최종 사용자/고객 자격 증명을 암호화하고 비밀번호를 저장할 때 해싱하십시오.

    만약 당신이 관리하는 데이터베이스에 최종 사용자/고객 자격 증명을 저장하고 있다면, 저장 상태에서 자격 증명을 암호화하고 비밀번호를 해시하세요. 대안으로, AWS는 Amazon Cognito 또는 동등한 AWS 파트너 솔루션과 같은 사용자 신원 동기화 서비스를 사용하는 것을 권장합니다.

  • IAM-012 - AWS에서 실행되는 애플리케이션이 다른 AWS 리소스에 액세스할 때 임시 자격 증명 사용

    AWS 리소스에 액세스할 때 임시 보안 자격 증명을 사용하십시오.

    • Amazon EC2 인스턴스 또는 AWS Lambda 함수와 같은 AWS 내 기계 ID에 대한 임시 보안 자격 증명을 얻기 위해 IAM 역할을 사용하십시오.
    • AWS 컴퓨팅 서비스에서 실행되는 모든 프로세스는 연결된 IAM 역할에서 가져온 임시 자격 증명을 사용하여 AWS API를 호출해야 합니다.

운영 보안

  • SECOPS-001 - 취약점 관리 수행.

    의존성 및 운영 체제의 취약점을 검사하고 패치하여 새로운 위협을 방지하기 위해 메커니즘과 빈도를 정의합니다. 정의된 일정에 따라 종속성과 운영 체제를 검사하고 패치합니다. 소프트웨어 취약점 관리는 시스템을 위협 요인으로부터 안전하게 유지하는 데 필수적입니다. 지속적인 통합/지속적인 전달 (CI/CD) 파이프라인 초기에 취약점 평가를 포함시키면 감지된 보안 취약점의 수정을 우선 순위로 지정할 수 있습니다. 이를 달성하기 위해 필요한 솔루션은 사용 중인 AWS 서비스에 따라 다릅니다. Amazon EC2 인스턴스에서 실행되는 소프트웨어의 취약점을 검사하려면 Amazon Inspector를 파이프라인에 추가하여 Inspector가 취약점을 감지하면 빌드가 실패하게 할 수 있습니다. 또한 OWASP Dependency-Check, Snyk, OpenVAS, 패키지 관리자 및 AWS 파트너 도구와 같은 오픈 소스 제품을 취약점 관리에 사용할 수 있습니다.

네트워크 보안

  • NETSEC-001 - 모든 Amazon EC2 보안 그룹에 대해 가장 허용되지 않는 규칙 구현.

    모든 Amazon EC2 보안 그룹은 최대한 제한해야 합니다. 최소한 다음을 수행하십시오:

    • 포트 22 또는 3389로 0.0.0.0/0 에서의 들어오는 트래픽을 허용하는 보안 그룹이 없어야 합니다

      ([CIS_5.0 5.4/ Security Control EC2.54 과 CIS_5.0 5.3/ Security Control EC2.53] 또는 [CIS_3.0 5.3/ Security Control EC2.54 과 CIS_3.0 5.2/ Security Control EC2.53])

    • 모든 VPC의 기본 보안 그룹은 모든 트래픽을 제한해야 합니다

      (CIS_5.0 5.5/Security Control EC2.2 또는 CIS_3.0 5.4/Security Control EC2.2)

  • NETSEC-002 - 공용 서브넷의 리소스 제한.

    VPC의 공용 서브넷에 리소스를 두지 마세요, 그것들이 공개 소스로부터 네트워크 트래픽을 받아야 하는 경우를 제외하고는. 공용 서브넷은 인터넷 게이트웨이로의 경로가 있는 라우트 테이블과 연결된 서브넷입니다.

백업 및 복구

  • BAR-001 - 자동 데이터 백업 구성.

    내구성 있는 저장 서비스로 정기적인 백업을 수행해야 합니다. 백업은 관리, 논리, 물리 오류 시나리오에서 복구할 수 있는 능력을 보장합니다. 주기적인 일정 또는 데이터 세트의 변경을 기반으로 자동으로 백업을 구성합니다. RDS 인스턴스, EBS 볼륨, DynamoDB 테이블 및 S3 객체는 모두 자동 백업을 위해 구성할 수 있습니다. AWS Backup, AWS Marketplace 솔루션 또는 타사 솔루션을 사용할 수도 있습니다. S3 버킷의 객체가 일회성-읽기-많은 (WORM) 경우, 이 요구 사항을 충족시키기 위해 객체 잠금과 같은 보상 제어를 사용할 수 있습니다. 고객이 데이터를 백업할 책임이 있는 경우, 문서에서 명확히 명시하고 파트너는 데이터를 백업하는 방법에 대한 명확한 지침을 제공해야 합니다.

  • BAR-002 - 정기적으로 데이터를 복구하여 백업 프로세스의 무결성을 확인합니다.

    정기적인 일정 및 클라우드 환경에 중요한 변경 후에 복구 테스트를 실행하여 복구 시간 목표 (RTO) 및 복구 지점 목표 (RPO)를 충족하는 백업 프로세스를 확인하십시오. 자세한 정보는 Getting Started - Backup and Restore with AWS를 참조하십시오.

탄력성

  • RES-001 - 복구 지점 목표(RPO)를 정의하십시오.

    시간을 기준으로 조직의 데이터 손실 허용치에 기반한 RPO를 정의하십시오.

    자가 평가 워크시트의 파트너 응답 열에 RPO를 입력하십시오.

  • RES-002 - 복구 시간 목표(RTO)를 수립하십시오.

    조직의 요구 사항과 기대치에 부합하는 RTO를 정의하십시오. RTO는 중단과 서비스 복구 사이에 조직이 허용할 최대 지연 시간입니다.

    자가 평가 워크시트의 파트너 응답 열에 RTO를 입력하십시오.

  • RES-004 - 탄력성 테스팅

    최소 12개월마다 정기적으로, 그리고 주요 업데이트 후에는, 항상 파트너사가 수립한 resiliency 목표를 충족했는지 복원력을 테스트해야 합니다. 복원력 테스트에는 최소한 가용 영역(AZ) 장애와 데이터 손실, 인스턴스 장애, 리전 장애 등 귀사의 아키텍처와 관련된 다른 장애 유형이 포함되어야 합니다. FTR 승인 전에 최소 1회 이상 성립된 요구 사항을 충족하는 복원력 테스트가 완료되어야 합니다.

    AWS Resilience Hub를 사용하여 워크로드가 복원력 목표를 충족하는지 확인할 수 있습니다. AWS Resilience Hub는 AWS Fault Injection Service(AWS FIS)와 협력하여 실제 장애 시나리오를 통한 결함 주입 실험을 제공하여 애플리케이션이 정의된 복원력 목표 내에서 복구되는지 검증할 수 있습니다. AWS Resilience Hub는 또한 CI/CD 파이프라인에 복원력 평가 및 테스트를 통합할 수 있는 API 작업을 제공하여 지속적인 복원력 검증을 가능하게 합니다. CI/CD 파이프라인에 복원력 검증을 포함하면 워크로드 기반 인프라의 변경으로 인한 복원력 저하를 방지할 수 있습니다.

    자체 평가 워크시트의 파트너 응답 열에 마지막 복원력 테스트 날짜를 입력해 주십시오.

  • RES-005 - 고객의 탄력성에 대한 책임을 소통하십시오.

    백업, 복구, 및 가용성에 대한 고객의 책임을 명확히 정의하십시오. 최소한, 제품 문서 또는 고객 계약은 다음을 포함해야 합니다:

    • 고객이 솔루션에 저장된 데이터를 백업하는 책임
    • 필요한 경우 데이터 보호를 위해 데이터를 백업하거나 제품에서 선택적 기능을 구성하는 지침
    • 제품의 가용성을 구성할 수 있는 고객의 옵션

  • RES-006 - 가용성 목표 및 업타임 서비스 수준 계약(SLA)을 충족하기 위해 제품을 설계하십시오.

    가용성 목표 또는 업타임 SLA를 공개 또는 사적으로 동의하는 경우, 이를 지원하기 위해 아키텍처와 운영 프로세스를 설계해야 합니다. 또한, 목표 또는 SLA를 달성하기 위해 필요한 구성에 대한 고객에게 명확한 지침을 제공하십시오.

  • RES-007 - 정전에 대한 고객 통신 계획을 정의하십시오.

    시스템 정전에 대한 고객에게 정보를 통신하는 계획을 수립하십시오. 통신은 비밀 계약(NDA) 하에 AWS로부터 제공된 데이터를 포함해서는 안됩니다.

Amazon S3 버킷 액세스

  • S3-001 - 모든 Amazon S3 버킷을 검토하여 적절한 액세스 수준을 결정합니다.

    공개 액세스가 필요한 버킷이 공개 읽기 또는 쓰기 액세스가 필요한지, 그리고 공개 액세스를 제어하기 위한 적절한 제어가 있는지 검토되었는지 확인해야 합니다. 액세스 권한을 할당할 때 principle of least privilege, an AWS best practice. 자세한 정보는 overview of managing access를 참조하십시오.

교차 계정 접근

  • CAA-001 - 고객 계정에 접근하기 위해 교차 계정 역할을 사용하십시오.

    Cross-account roles는 AWS 파트너가 고객을 위해 저장해야 하는 민감한 정보의 양을 줄입니다.

  • CAA-007 - 최소한의 필요 권한으로 교차 계정 역할을 생성하기 위한 지침이나 자동 설정 메커니즘(예: AWS CloudFormation 템플릿)을 제공하십시오.

    고객 계정에서 교차 계정 접근을 위해 생성된 정책은 principle of least privilege를 따라야 합니다. AWS 파트너는 역할 정책 문서 또는 자동 설정 메커니즘(예: AWS CloudFormation 템플릿)을 제공하여 고객이 최소한의 필요 권한으로 역할을 생성할 수 있도록 해야 합니다. 자세한 내용은 AWS Partner Network (APN) blog posts를 참조하십시오.

  • CAA-002 - 고객 계정에 접근하기 위해 교차 계정 역할과 함께 외부 ID를 사용하십시오.

    외부 ID를 사용하면 역할을 가정하는 사용자가 그들이 작업하는 상황을 주장할 수 있습니다. 또한 계정 소유자가 특정 상황에서만 역할을 가정하도록 허용하는 방법을 제공합니다. 외부 ID의 주요 기능은 confused deputy 문제를 해결하고 방지하는 것입니다.

  • CAA-004 - 외부 ID에 대해 고객이 제공한 것이 아닌 생성한 값을 사용하십시오.

    IAM 역할을 사용하여 교차 계정 접근을 설정할 때, 고객이 제공한 것이 아닌 생성한 값을 외부 ID에 사용해야 교차 계정 역할 설정의 무결성을 보장합니다. 파트너가 생성한 외부 ID는 악의적인 당사자가 고객의 설정을 가장할 수 없게 하고 모든 고객 간의 고유성과 형식 일관성을 강제합니다. 만약 오늘날 외부 ID를 생성하고 있지 않다면, 고객이 교차 계정 역할을 설정하는 데 사용하는 외부 ID에 대해 무작위 고유 값(예: Universally Unique Identifier)을 생성하는 프로세스를 구현하는 것을 추천합니다.

  • CAA-005 - 교차 계정 역할을 사용하는 고객이 역할을 가정할 수 있는 조건을 명시하십시오.

    고객 계정에서 교차 계정 역할을 사용할 때, 역할을 가정할 수 있는 조건을 명시해야 합니다. 이는 고객이 역할을 가정하는데 필요한 권한과 절차를 이해할 수 있도록 도움을 줍니다. 자세한 내용은 cross-account roles를 참조하십시오.

  • CAA-006 - 모든 외부 ID가 고유한지 확인하세요.

    사용되는 외부 ID는 모든 고객 간에 고유해야 합니다. 다른 고객에 대해 외부 ID를 재사용하면 혼란스러운 대리인 문제를 해결하지 못하며, 고객 A가 고객 B의 역할 ARN과 외부 ID를 사용하여 고객 B의 데이터를 볼 수 있는 위험이 있습니다. 이를 해결하기 위해, 우리는 고객이 교차 계정 역할을 설정하는데 사용하는 외부 ID에 대해 랜덤한 고유 값, 예를 들어 Universally Unique Identifier를 생성하는 과정을 구현하는 것을 권장합니다.

  • CAA-003 - 고객이 제공한 IAM 자격증명의 역사적인 사용을 폐지하세요.

    귀하의 애플리케이션이 교차 계정 접근을 위한 정적 IAM 자격증명 사용에 대한 레거시 지원을 제공한다면, 애플리케이션의 사용자 인터페이스 및 고객 문서는 이 방법이 더 이상 사용되지 않는다는 것을 명확히 해야 합니다. 기존 고객들은 cross-account role based-access로 전환하도록 장려되어야 하며, 새 고객에 대한 자격증명 수집은 비활성화되어야 합니다.

민감한 데이터

  • SDAT-001 - 민감한 데이터를 식별하십시오 (예: 개인 식별 정보 (PII) 및 보호된 건강 정보 (PHI)).

    데이터 분류를 통해 어떤 데이터를 보호해야 하고 어떻게 보호해야 하는지 결정할 수 있습니다. 작업 부하와 처리하는 데이터를 기반으로 일반 공공 지식이 아닌 데이터를 식별합니다.

  • SDAT-002 - 모든 민감한 데이터를 저장할 때 암호화하십시오.

    암호화는 데이터가 도난당하거나 전송되는 네트워크가 침해되더라도 민감한 데이터의 기밀성을 유지합니다.

  • SDAT-003 - VPC 외부로 민감한 데이터를 전송할 때 암호화가 있는 프로토콜만 사용하십시오.

    암호화는 전송되는 네트워크가 침해되더라도 데이터 기밀성을 유지합니다.

규제 준수 검증 프로세스

  • RCVP-001 - 모든 필수 준수 표준이 충족되도록 프로세스를 수립하십시오.

    제품이 특정 준수 표준을 충족한다고 광고하면 내부 프로세스를 통해 준수를 보장해야 합니다. 준수 표준의 예로는 Payment Card Industry Data Security Standard (PCI DSS) PCI DSS, Federal Risk and Authorization Management Program (FedRAMP)FedRAMP, 및 U.S. Health Insurance Portability and Accountability Act (HIPAA)HIPAA가 있습니다. 적용 가능한 준수 표준은 솔루션이 저장하거나 전송하는 데이터 유형 및 지원하는 지리적 지역과 같은 다양한 요소에 의해 결정됩니다.
